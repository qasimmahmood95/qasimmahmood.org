Background

I'm a Senior Software Development Engineer in Test with more than seven years in regulated fintech, currently focused on digital asset custody. Most of my work is on release gates, contract coverage and the compliance-critical test suites a regulated platform depends on.

My specialism is building automation frameworks from nothing: web with Playwright, mobile with Maestro and Appium, load with k6. I drop below the UI wherever that gives better coverage, from simulating biometric payloads over API to validating the compliance flows, such as the Travel Rule, that regulators audit.

I also lead an internal AI advocacy initiative, helping engineers adopt AI tooling without weakening the controls a regulated firm has to maintain.

I came to software from engineering of a different kind: an MEng in Aerospace Materials from Imperial College London, then a graduate year examining superalloys bound for jet engines. Metallurgy and software quality turn out to have the same core question, which is how things fail under stress.

✓ 02 / approach

How I test

01 / below the UI where it counts

UI checks prove the journey works; the risky logic lives in the services underneath. When hardware makes a test flaky, I take the hardware out of the loop, as with simulating biometric payloads over API instead of driving physical devices.

02 / compliance gates as code

Rules that matter to regulators, such as the Travel Rule, become executable checks in the release pipeline rather than lines in a test plan. If a change would breach one, it does not ship.

03 / frameworks are products

I've built automation from zero four times across web, mobile, API and load. A framework earns its keep when other engineers can write, run and trust tests without me in the room.

04 / ai with guardrails

I use AI tooling daily and lead its adoption at work. The point is speed without surrendering the controls a regulated firm depends on; adoption and caution are not opposites.
